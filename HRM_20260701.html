背景 · CONTEXT

一個集團,多公司、多國,需求各不相同

兆聯實業旗下含多家子公司與分公司,各自需要「獨立」的 HRM。其中橫跨三國,勞動法規與計薪方式是根本性的差異,而非微調。
HRM · 02 / 18
GROUP · COMPANIES · JURISDICTIONS

三個層次的「不同」

①公司層級不同:組織、簽核、成本中心各異 — 屬些許差異,可用設定吸收。
②作業習慣不同:班別、假別、加項名稱 — 屬中度差異,可用參數表與擴充明細吸收。
③法規與計薪不同:台/美/新的勞動法、社保、稅制 — 屬根本性差異,需在地化計薪引擎。
Key Question
「獨立」不等於「各做一套」。真正要決定的是:哪些共用、哪些隔離、隔離到什麼程度。
問題定義 · TWO DECISIONS

要拍板的,其實是兩個架構決策

資料庫層與模組功能層各有一個「共用 ↔ 分離」的光譜。兩者並非二選一,而是要找到對的切割點。

DECISION FRAMEWORK
關鍵釐清 · COMMON PITFALL

先釐清:EntityID ≠ 公司識別

現有規格中 EntityID 是「系統實體」(HRM/LIMS/MAT),固定為 'hrm'。若把公司多租戶綁在它身上,會破壞與 sys 模組的對齊。公司要靠另一個識別維度。
HRM · 04 / 18
IDENTITY DIMENSIONS

為什麼這個分辨很重要

EntityID 在你既有架構是系統層的鍵,所有 hrm* 表固定 'hrm',對齊 sys.sysOrg / sysUser。把公司塞進去 = 同一個鍵兼兩種語意,日後組織同步與權限全亂。

公司用 CompanyCode(或由員工現職部門推導),掛在交易表上做列級區隔。

國別再獨立一欄 PayrollCountry,專門決定計薪邏輯走哪一包 — 與「公司」解耦,因為一家公司也可能跨國派駐。

結論
三個維度各司其職:系統=hrm、公司=CompanyCode、國別=PayrollCountry。下面的架構都建立在這個分辨上。
決策一 · DATABASE

資料庫:共用 Table vs 每公司一組 Table

八個構面逐項對照。對 HRM 而言,殺手級因素是「集團合併報表」與「外派跨公司」——這兩者強烈偏向共用。
HRM · 05 / 18
對比構面🟦 共用 Table + 公司識別欄🟥 每公司一組 Table
Schema 維護✓ 改一次全公司生效✗ N 份要同步,易 drift
集團合併報表✓ 一個 WHERE 跨公司彙總✗ N 份 UNION 或另建資料倉儲

資料隔離強度△ 邏輯隔離,需 RLS 兜底✓ 物理隔離,洩漏面最小
跨境資料落地✗ 全混一庫,難符 PDPA / 落地✓ 可分地域庫存放
效能 / 維運獨立△ 單一大表,靠索引;備份一起✓ 各公司獨立備份 / 維護窗口
建議
以共用 Table 為主,搭配 RLS 把「漏 WHERE 就外洩」的風險拉回 DB 引擎層;僅跨境機敏欄位做局部區域庫。
🎯
物理分離的唯一正當理由
跨境法遵/資料落地,且僅針對「特定公司 × 特定機敏欄位」,不該為它切整個庫。
決策二 · MODULE FUNCTION

模組功能:共用功能 vs 一公司一模組

差異是「些許/中度」就該用設定吸收,不該複製程式。唯有計薪因三國法規根本不同,才需在特定領域插拔。

程式維護✓ 單一 codebase,修一次全到位✗ N 份程式,重複工 + 分歧

關鍵原則:差異用「設定」解決,不用「複製程式」解決
假別、班別、加項名稱、簽核關卡的差異,該進參數表與擴充明細;一旦走「一公司一模組」,等於用最貴的方式處理本可參數化的差異。例外只有計薪——見下一頁。
核心約束 · THREE JURISDICTIONS

三國計薪,差異有多大?幾乎沒有交集

社保提撥、加班規則、稅務、年度文件——每一項在台/美/新都是不同的法定機制,無法用同一條公式表達。

Implication
這不是「些許不同」可參數化的範圍。硬塞成單一引擎 = 滿是 if-else 的不可維護怪物。正解是:三個獨立的在地化計薪引擎。
領域切割 · DOMAIN PARTITION

差異是領域性的,不是公司性的

把八個模組攤開:只有薪資與其下游(社保/稅/假別/加班)真正分國;其餘 80% 跨國長得幾乎一樣,維持共用。

SHARED SKELETON · COUNTRY RULES

設計要點

✓主檔加 PayrollCountry 欄:每筆薪資知道自己屬哪一國,分派器據此決定引擎。
✓hrmSalaryItem 泛用化:不為各國各開一張表,而是一張明細表、項目類型由各國定義,合併彙總零障礙。
✓規則表以國別為 key:勞健保/FICA/CPF 級距各自一套,互不干擾,修法只動該國。
效果
外派雙軌只是同一員工產生兩筆 hrmSalary(TW + US),寫進同一張表,中央一個查詢即見。

安全與法遵 · SECURITY

機敏資料:RLS 兜底 + 區域庫落地

共用 Table 對 HR 機敏資料(薪資 L2、身份證 L3)的最大顧慮,用 RLS 把隔離從應用層拉回引擎層;跨境落地需求由區域庫處理。
HRM · 15 / 18
ROW-LEVEL SECURITY

兩道防線

① RLS 引擎層隔離
SQL Server 安全原則依連線身分自動附加公司/國別過濾。把「漏 WHERE 就外洩」的風險從人為失誤拉回資料庫強制執行。
② 區域庫(跨境落地)
美/新的薪資機敏資料(SSN、稅務、薪資)若有落地要求,放區域庫。邊界正好和「計薪包」對齊——只在這層切,人事主檔仍中央集中。
分級對應
身份證號 L3 加密、薪資/銀行帳號 L2 遮罩,匯出留稽核軌跡——延續 B00 既有敏感分級原則。

DECISION 1 · 資料庫
共用 Table + 公司識別欄 + RLS
薪資領域用「共用骨架(hrmSalary/hrmSalaryItem)+ 各國規則表」。僅跨境機敏欄位做局部區域庫,不為它切整個庫。
DECISION 2 · 模組功能
共用核心 + 計薪包(在地化)
其餘模組單一 codebase;計薪引擎做成一介面三實作(TW/US/SG)。差異用設定吸收,不複製程式。
✓ 為什麼這是中道
避免「一國一套」的重複與外派斷裂,也避免「單一引擎」的不可維護。外派雙軌、集團合併報表全部活著。
✓ 唯一反向選擇
若某子公司未來要分拆出售或受監管要求完全隔離,該公司可從一開始就放獨立庫;其餘維持共用。
